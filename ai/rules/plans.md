# Plans rules

## Purpose

Define mandatory constraints for naming, structure, and storage of implementation plans.

These rules ensure consistency, traceability, and scalability across all plans.

## Scope

These rules apply to:

- implementation plans generated by AI
- manually created plans
- plans under `ai/plans/`
- any artifact representing a planned implementation

## Core principles

- Plans must be consistently structured.
- Plans must be traceable to requirements.
- Naming must be predictable and stable.
- Plans must scale across domains and features.
- Plans must not be duplicated.
- Plans must follow a single source of truth.

## Mandatory rules

- Plans must be stored under `ai/plans/`.
- Plans must follow this structure:

  `ai/plans/<area>/<requirement-name>.md`

- `<area>` must represent a meaningful functional or technical domain.
- `<requirement-name>` must match the requirement file name.
- Plan names must:
  - be lowercase
  - use hyphen-separated words
  - avoid spaces
  - avoid non-standard abbreviations
- Plans must not be stored outside `ai/plans/`.
- Duplicate plans are not allowed.
- Each plan must reference its requirement.
- Each requirement must have at most one active plan.
- Plans must be reviewed before implementation.
- Plans must not be modified without re-review.
- Plans must be traceable to:
  - requirement
  - impacted areas
  - decisions taken

## Additional considerations

- Plan evolution must be tracked via version control.
- Plans should remain stable once approved.
- Historical versions must not be overwritten without traceability.
- Plans may include optional metadata fields such as:
  - requirement reference
  - area
  - status (draft | approved | implemented)
  These fields are optional and must not replace the plan structure.

Related commands:

- `ai/commands/create-implementation-plan.md`
- `ai/commands/review-plan.md`

## Rule priority

If there is a conflict between plan structure and naming preferences, these rules take priority.

If generated plans do not comply with these rules, they must be corrected.

Plans must never override requirements, approved decisions, or architecture rules.
